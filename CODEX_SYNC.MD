# Codex 双电脑接力规则

本项目经常在办公室电脑和家里电脑之间切换。为了减少 Git 冲突和重复排查，所有 Codex 会话按下面方式接力。

## 开工前

1. 先看本地状态：`git status --short`。
2. 如果本地没有未完成工作，先执行 `git pull --rebase --autostash`。
3. 如果发现未提交改动，先判断是否是上一台电脑留下的工作：
   - 已经在 Git 上的，以远端为准继续。
   - 本地未提交但不清楚来源的，先读 `CODEX_HANDOFF_CURRENT.MD`，不要直接覆盖。
   - 只属于微信开发者工具的本机配置变化，例如 `project.config.json`，默认不要当成业务改动处理。

## 收工前

1. 能完成的工作尽量提交并推送，让另一台电脑直接 `git pull` 接上。
2. 如果工作还没完成，不强行提交半成品；更新 `CODEX_HANDOFF_CURRENT.MD`，写清楚：
   - 当前做到哪一步。
   - 哪些文件是相关改动。
   - 已验证什么。
   - 下一台电脑继续做什么。
3. 涉及云服务器的工作，必须写清楚服务器是否已经拉取、重启、验证。

## Git 习惯

- 默认在当前主分支上做小步提交，避免办公室和家里电脑长期各自堆未提交改动。
- 每次切换电脑前优先保证远端 Git 是最新的。
- 不提交 `.env`、密钥、Betfair token、`server/data/`、云端临时传输文件。
- 不为了同步而提交明显只属于本机工具的配置噪音。

## 变更日志精简规则

`CODEX_CHANGELOG.MD` 只记录以后排查有价值的事项：

- 新功能、页面结构、用户流程变化。
- 比赛数据库、赛程、预测、复盘、排名、Betfair 数据流程变化。
- 云函数、轻量服务器、远程 API、用户数据结构变化。
- 需要重新部署、重新上传、重新审核的事项。

以下内容可以不写入 changelog，除非它们引发线上问题或和大改动一起提交：

- 小样式微调。
- 文案错别字。
- 单个很小的 bug 修复。
- 本机开发工具配置变化。
- 临时调试、检查命令、无行为变化的整理。

写 changelog 时用短格式即可：

```text
## YYYY-MM-DD HH:mm - 一句话主题
- 目的：
- 涉及：
- 影响：
- 验证：
- 部署：
```
